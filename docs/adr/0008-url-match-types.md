# 규칙 URL 필터는 매치 방식을 고른다 (도메인/포함/시작/정규식)

정규식 한 칸은 비전문 사용자에게 장벽이고, 모든 스코프가 regex 규칙 한도(1,000)를 소모한다. 규칙의 URL 필터에 매치 방식을 추가한다: `urlMatchType?: 'domain' | 'contains' | 'prefix' | 'regex'`.

- **컴파일 매핑** — 정규식만 `regexFilter`, 나머지는 DNR의 비정규식 `urlFilter` 문법으로 내려간다:
  - `domain` → `||패턴` (서브도메인 포함 도메인 앵커)
  - `contains` → `패턴` (URL 부분 문자열)
  - `prefix` → `|패턴` (URL 시작 앵커)
  - `regex` → `regexFilter` (기존 동작)
- **한도 이득**: 비정규식 방식은 regex 규칙 카운터를 소모하지 않는다 — 스코프 규칙 용량이 사실상 총량 한도(5,000)까지 확장된다.
- **하위 호환**: `urlMatchType` 부재 + `urlFilter` 존재 = `regex`(기존 데이터 의미 보존). 새 규칙의 폼 기본값은 `contains`(가장 직관적). 저장 시 `urlFilter`가 비면 두 필드 모두 벗긴다.
- **검증**: `regex` 방식만 `isRegexSupported` 저장 시점 검증. 비정규식 패턴도 길이 한도(프로필 조인과 동일)를 공유하고 초과 시 방출 금지+경고(ADR 0007과 동일한 원칙).

트레이드오프: 스키마 필드 하나 추가. DNR urlFilter 문법 특수문자(`*`, `|`, `||`, `^`)가 패턴에 섞이면 사용자 의도와 다르게 해석될 수 있다 — 문서화하되 이스케이프하지 않는다(고급 사용 여지).
