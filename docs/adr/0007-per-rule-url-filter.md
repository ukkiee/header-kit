# 규칙은 자기 URL 필터를 가질 수 있다

프로필 수준 URL 필터만으로는 "이 헤더는 이 URL에만"이라는 가장 흔한 의도를 규칙 단위로 표현할 수 없다. 스키마에 선택 필드를 추가한다: 헤더 4종(request/response/cookie/set-cookie)과 CSP 수정이 `urlFilter?: string`(regex)을 가진다. 리다이렉트는 제외 — 자신의 pattern이 이미 URL 매처다.

- **컴파일 의미론**: 규칙에 urlFilter가 있으면 그 규칙의 regexFilter는 자기 필터다 — 프로필 URL 필터 조인을 **대체**한다(그 규칙에 한해). 프로필의 나머지 차원(리소스 타입·메서드·initiator·탭·exclude·시간)은 그대로 상속한다. 프로필 URL 필터는 자기 필터가 없는 규칙에만 적용된다.
- **한도**: 규칙 자체 필터가 조인 한도(2KB)를 넘으면 규칙을 **방출하지 않고** regex-too-long 경고를 낸다 — 스코프를 조용히 넓히는 것(전 URL 적용)이 더 위험하기 때문.
- **저장 시점 검증**: redirect pattern·URL 필터와 동일하게 `isRegexSupported`로 검증 — 무효 regex는 저장 자체가 거부된다(import 포함).
- **UI**: 규칙 폼의 URL 필터 입력이 이 필드를 편집한다(ADR 0006의 프로필 스코프 편집을 대체). 요약 행의 `스코프 → 효과` 프리픽스도 자기 필터를 보여준다. 프로필 수준 URL/기타 필터는 필터 탭에 남는다.
- 스키마는 additive(선택 필드) — 기존 데이터·export 무이주 호환.

트레이드오프: 같은 스코프를 여러 규칙에 반복 입력할 수 있다(프로필 필터가 그 중복을 흡수하는 도구로 남는다). regex 규칙 수 한도(1,000) 소모가 늘 수 있다 — 경고 체계가 기존대로 알린다.
