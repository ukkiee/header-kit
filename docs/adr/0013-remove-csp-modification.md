# CSP 수정 종류를 제거한다

`csp` 수정 종류(응답 `Content-Security-Policy` 헤더를 디렉티브 합성으로 set 하던 것)를 제품에서 제거한다. 규칙 종류는 헤더 계열(request/response header, cookie, set-cookie)과 redirect만 남는다.

- **맥락**: CSP 수정은 개발 중 사이트의 CSP를 느슨/엄격하게 덮어쓰는 니치 용도였다. 실사용 수요가 낮고, 규칙 모델 표면(디렉티브 편집기 UI·검증·컴파일·마이그레이션 분기)을 유지하는 비용이 그에 비해 크다. 조건 체계를 규칙 속성으로 모은 ADR 0010이 "니치 기능은 쳐낸다(탭/창 피커·제외 URL 소실)"는 선례를 이미 세웠다 — 이 결정은 그 방향의 연장이다.
- **결정**: `CspModification`·`CspDirective`와 `'csp'` union 멤버, 그에 딸린 컴파일(`emitCspRule`)·검증(`directives` 필수)·UI(Type 옵션·디렉티브 편집기·kind-label·rule-summary)·i18n(`modCsp`·`ariaCspDirective*`)를 전부 걷어낸다.
- **마이그레이션 — 데이터 손실을 감수한다**: 이미 저장된 `kind:'csp'` 규칙은 **로드와 import 양 진입점에서 조용히 버린다**(안내 없음). 두 경로가 같은 처리라 코드가 단순하고, 로드 검증이 csp를 만나 전체 상태를 리셋하는 일을 막는다. 현재 v0.1.0이라 실사용자 영향은 사실상 없다. ADR 0010이 소실 종류를 마이그레이션에서 버린 것과 같은 형태다.
- **경계**:
  - `csp_report`는 **리소스 타입**(declarativeNetRequest)이지 이 CSP 수정과 무관하다 — 리소스 종류 조건 칩에 그대로 남는다. 건드리지 않는다.
  - import는 로드와 달리 명시적 행위지만, 기존 "소실 종류 안내" 패턴을 CSP엔 쓰지 않기로 했다(사용자 결정) — 로드와 동일하게 조용히 버려 두 경로를 일치시킨다.
- **트레이드오프**: CSP를 쓰던 사용자는 그 기능을 잃고 기존 규칙이 소실된다. 대신 규칙 종류가 하나 줄어 멘탈 모델·검증·컴파일·마이그레이션이 간결해진다.
