# ui-refine — 아이콘·툴팁, Base UI 일원화, 드래그 재정렬, 저장 검증, 조건 배지, motion

**Triage:** ready-for-agent

## Problem Statement

팝업 UI가 기능적으로는 완성됐지만 상호작용 품질이 고르지 않다. 편집·삭제·복원·펼치기가 텍스트 버튼이라 행마다 시각적 소음이 반복되고, 아이콘이 없는 곳은 의미 전달이 느리다. 이름이 빈 규칙이 저장을 통과한 뒤에야 "건너뛰었습니다" 경고로 나타나 사용자가 원인을 역추적해야 한다. 프로필 순서 변경이 ⋯ 메뉴 안에 숨어 있어 여러 칸 이동이 번거롭다. 규칙에 붙인 Condition이 목록에서 "조건: 1"이라는 개수로만 보여 무슨 조건인지 열어보기 전엔 알 수 없다. 종류 선택 폼에는 어긋난 라벨(요청 쿠키의 '헤더 이름'), 무의미한 컨트롤(append 불가 헤더의 모드 선택), 매치 방식과 맞지 않는 placeholder가 남아 있다. 네이티브 셀렉트 팝업은 OS 스타일이라 미니멀 UI와 이질적이고, 칩 그룹에는 엉뚱한 영역 호버에 첫 칩이 반응하는 버그가 있다. 상태 변화(행 추가·삭제, 화면 전환)가 즉시 튀어 변화 인지가 어렵고, 환경설정에는 불필요한 단축키 안내문이 있고 기본 자동완성 사전은 보이지 않는다.

## Solution

상호작용 계층을 한 번에 정제한다. 반복 액션은 전부 lucide 아이콘 + 툴팁으로 바꾸고, 저장은 "저장되면 반드시 동작하는 규칙"만 통과시키며, 프로필 순서는 사이드바에서 드래그(키보드 포함)로 바꾼다. Condition은 값 배지 줄로 한눈에 보이게 하고, 종류 라벨·placeholder·컨트롤 노출을 종류별 의미에 맞게 다듬는다. UI 프리미티브는 Base UI로 일원화해 셀렉트·칩·패널·툴팁·토스트의 스타일과 접근성 시맨틱을 통일하고(칩 호버 버그도 구조적으로 해소), motion으로 행 추가·삭제·폼 열림·화면 전환·마이크로 인터랙션에 일관된 모션 언어를 입힌다. 규칙 삭제는 토스트로 실행 취소를 제공하고, 빈 프로필에는 첫 규칙 추가를 유도한다.

## User Stories

1. As a 사용자, I want 편집·삭제가 텍스트 대신 아이콘 버튼으로 보이기를, so that 행이 간결해지고 액션을 시각적으로 빠르게 찾을 수 있다
2. As a 사용자, I want 아이콘 버튼에 마우스를 올리면 툴팁으로 이름이 보이기를, so that 아이콘만으로 의미가 불확실할 때 확인할 수 있다
3. As a 키보드 사용자, I want 아이콘 버튼이 포커스될 때도 툴팁이 보이기를, so that 마우스 없이도 같은 정보를 얻는다
4. As a 사용자, I want 헤더 이름이 빈 채로 저장을 누르면 그 자리에서 오류가 보이고 저장이 막히기를, so that 동작하지 않는 규칙이 만들어지지 않는다
5. As a 사용자, I want 요청 쿠키의 쿠키 이름, CSP의 디렉티브 이름, Redirect의 패턴·치환이 빈 경우도 같은 방식으로 막히기를, so that 어떤 종류든 저장된 규칙은 반드시 동작한다
6. As a 사용자, I want 응답 쿠키의 빈 값은 여전히 저장되기를, so that 서버 Set-Cookie 차단 사용례가 유지된다
7. As a 사용자, I want 검증 오류가 해당 입력 옆에 인라인으로 보이기를, so that 무엇을 고쳐야 하는지 즉시 안다
8. As a 사용자, I want 사이드바에서 프로필을 드래그해 순서를 바꾸기를, so that 메뉴를 거치지 않고 한 동작으로 우선순위를 조정한다
9. As a 키보드 사용자, I want 그립에 포커스해 Space로 집고 화살표로 옮기기를, so that 드래그 없이도 순서를 바꿀 수 있다
10. As a 사용자, I want ⋯ 메뉴가 복제·삭제만 남기를, so that 메뉴가 짧고 명확해진다
11. As a 사용자, I want 규칙 행에서 조건이 [POST] [main_frame] 같은 값 배지로 보이기를, so that 열어보지 않고도 어떤 조건인지 안다
12. As a 사용자, I want 제외 도메인·만료 시각 배지가 다른 조건과 구별되는 표기를 갖기를, so that 배지만 보고 조건의 방향(적용/제외/시한)을 안다
13. As a 사용자, I want 조건 없는 행은 지금과 같은 높이를 유지하기를, so that 목록 밀도가 나빠지지 않는다
14. As a 사용자, I want 종류 선택지에 'Set-Cookie' 대신 '응답 쿠키'가 보이기를, so that 요청 쿠키와 대칭인 멘탈 모델로 고를 수 있다
15. As a 기술 사용자, I want 행 배지는 SET-COOKIE 그대로이기를, so that 실제 수정되는 헤더가 무엇인지 정확히 안다
16. As a 사용자, I want append가 불가능한 요청 헤더에서 모드 선택이 아예 보이지 않기를, so that 선택할 수 없는 컨트롤에 혼란되지 않는다
17. As a 사용자, I want 요청 쿠키 폼의 이름 입력 라벨이 '쿠키 이름'이기를, so that 무엇을 넣는 칸인지 정확히 안다
18. As a 사용자, I want URL 필터의 placeholder가 매치 방식(정규식/포함/도메인/시작)에 맞는 예시이기를, so that 방식별 올바른 입력 형태를 즉시 안다
19. As a 사용자, I want 리소스 종류·메서드 칩에서 실제 올린 칩만 호버 반응하기를, so that 엉뚱한 첫 칩이 반응하는 혼란이 없다
20. As a 사용자, I want 백업·환경설정 패널의 펼치기/접기가 텍스트 대신 방향 아이콘이기를, so that 패널 헤더가 간결해진다
21. As a 사용자, I want 백업 복원 버튼이 아이콘+툴팁이기를, so that 목록이 간결하면서 의미는 유지된다
22. As a 사용자, I want 환경설정에서 단축키 안내문이 사라지기를, so that 화면에 실행 불가능한 안내가 없다
23. As a 사용자, I want 기본 자동완성 헤더 사전이 환경설정에 보이되 제거 불가이기를, so that 어떤 제안이 기본 제공되는지 알고 실수로 없앨 수 없다
24. As a 사용자, I want 내가 추가한 자동완성 항목만 X로 제거 가능하기를, so that 내 항목과 기본 항목이 구별된다
25. As a 사용자, I want 규칙 추가·삭제 시 행이 자연스럽게 나타나고 사라지기를, so that 무엇이 바뀌었는지 시각적으로 따라갈 수 있다
26. As a 사용자, I want 폼과 조건 disclosure가 부드럽게 열리고 닫히기를, so that 레이아웃 점프 없이 맥락이 유지된다
27. As a 사용자, I want 프로필↔백업↔환경설정 화면 전환에 짧은 전환 효과를, so that 화면이 바뀌었음을 인지하기 쉽다
28. As a 사용자, I want 스위치·칩 같은 작은 컨트롤에도 일관된 모션 반응을, so that 앱 전체가 하나의 모션 언어로 느껴진다
29. As a 모션 민감 사용자, I want 시스템의 reduced-motion 설정이 존중되기를, so that 애니메이션 없이 같은 기능을 쓸 수 있다
30. As a 사용자, I want 폼이 열리면 첫 입력에 포커스가 가기를, so that 바로 타이핑을 시작할 수 있다
31. As a 사용자, I want Esc로 폼이 닫히고 Cmd/Ctrl+Enter로 저장되기를, so that 키보드만으로 편집을 마칠 수 있다
32. As a 사용자, I want 규칙을 삭제하면 토스트에서 실행 취소할 수 있기를, so that 실수 삭제를 즉시 되돌릴 수 있다
33. As a 사용자, I want 실행 취소된 규칙이 원래 위치·상태로 돌아오기를, so that 복구가 완전하다
34. As a 사용자, I want 편집·삭제 아이콘이 행 호버·포커스 시에만 보이기를, so that 목록이 읽기 모드에서 최대한 조용하다
35. As a 사용자, I want 규칙 0개 프로필에 안내와 규칙 추가 유도가 보이기를, so that 다음에 무엇을 해야 할지 안다
36. As a 사용자, I want 셀렉트 팝업이 앱과 같은 스타일이기를, so that OS 네이티브 팝업의 이질감이 없다

## Implementation Decisions

- **Base UI 전면 일원화 (ADR 0011)**: 셀렉트(네이티브→Base UI Select), 칩 그룹(커스텀→Toggle/ToggleGroup), 접이식 패널(커스텀→Collapsible), 입력·체크박스·필드(커스텀→Base UI Input/Checkbox/Field/Fieldset), 신규로 Tooltip과 Toast를 도입한다. 기존 Base UI 사용처(메뉴·토글 스위치·대형 편집기 Dialog)는 유지한다.
- **칩 호버 버그의 구조적 해소**: 원인은 칩 버튼들이 라벨 요소 안에 래핑되어 브라우저가 라벨 호버를 연결된 첫 컨트롤에 전파하는 것. ToggleGroup 교체 시 캡션과 컨트롤 그룹을 분리해 이 구조 자체를 제거한다.
- **저장 검증은 core의 순수 함수**: Modification을 받아 종류별 누락 필수 필드 목록을 반환하는 순수 함수를 core에 둔다 — 요청/응답 헤더·요청 쿠키는 이름, CSP는 디렉티브 이름 최소 1개, Redirect는 패턴과 치환. 응답 쿠키는 빈 값이 유효(차단 사용례)라 필수 없음. 폼 Save는 이 함수 결과가 비어야 통과하고, 누락 필드에 aria-invalid와 인라인 오류를 표시한다. Compile의 빈 이름 경고는 import·레거시 데이터 방어선으로 유지한다.
- **드래그 재정렬**: dnd-kit(core+sortable)을 도입, 사이드바 프로필 항목에 그립을 두고 Pointer·Keyboard 두 센서를 켠다. KeyboardSensor는 sortable 좌표 전략(sortableKeyboardCoordinates)을 사용하고, 집기·이동·드롭 동안 포커스는 그립에 유지되며 Esc는 집기를 취소해 원래 순서로 되돌린다. 드롭은 기존 move-profile 명령으로 귀결된다(상태 전이 로직 변경 없음). ⋯ 메뉴의 위/아래 이동 항목은 제거하고 복제·삭제만 남긴다. (plan r1 R-2 반영)
- **조건 배지 줄**: 규칙 읽기 요약(ruleView)이 조건 배지 데이터 목록을 반환하도록 확장하고, 행은 요약 아래 둘째 줄에 값 배지를 렌더한다. 차원이 구별되는 표기를 쓴다 — 제외 도메인은 부정 접두, 만료는 시계 아이콘+시각, 나머지는 값 그대로. 조건 없으면 배지 줄 자체가 없어 행 높이가 불변한다. 요약 문자열의 'Conditions: n' 표기는 배지 줄로 대체된다.
- **응답 쿠키 명명**: 종류 라벨은 '응답 쿠키'/'Response cookie'(요청 쿠키와 대칭), 행 배지는 프로토콜 토큰 SET-COOKIE 유지, 스키마 kind 값 불변. CONTEXT.md 용어집에 반영 완료.
- **폼 다듬기**: append 허용 목록 밖 요청 헤더는 모드 셀렉트를 숨긴다(비활성화가 아니라 미노출). 요청 쿠키의 이름 라벨은 '쿠키 이름'. URL 필터 placeholder는 매치 방식별로 분기한다.
- **motion 전면 적용**: LazyMotion으로 초기 번들을 억제하고, 행 추가/삭제(enter/exit), 폼·disclosure 열림, 레일 화면 전환, 스위치·칩 마이크로 인터랙션까지 일관 적용한다. prefers-reduced-motion을 존중하고, 드래그 애니메이션은 dnd-kit 자체 transform에 위임한다(이중 애니메이션 금지).
- **아이콘·툴팁**: 아이콘은 기존 채택(lucide) 유지 — 편집·삭제·복원·패널 펼침 등 반복 액션 전부. 모든 아이콘 버튼은 aria-label과 툴팁 텍스트를 같은 카탈로그 키에서 공유한다.
- **행 액션 호버 표시**: 편집·삭제 아이콘은 행 호버·포커스(focus-within) 시에만 표시한다.
- **규칙 삭제 실행 취소**: 삭제는 즉시 실행하되 토스트로 Undo를 제공한다. Undo 스냅샷은 삭제 시점에 {Modification 원본, 원래 인덱스, 해당 materialized 값}의 3요소를 보관하고, 단일 명령이 이를 원자적으로 복원한다 — 일반 추가 경로를 타지 않으므로 Placeholder 규칙도 재실체화 없이 삭제 전과 동일한 실행 상태(materialized 값·컴파일 결과)로 돌아온다. 프로필 삭제의 기존 2단 확인은 유지한다. (plan r1 R-1 반영)
- **빈 상태 CTA**: 규칙 0개 프로필 본문에 안내 문구와 규칙 추가 유도를 표시한다.
- **환경설정**: 단축키 안내문 제거. 표준 헤더 사전 전체를 제거 불가 pill로 노출하고 사용자 항목만 X 제거를 유지한다.
- **폼 키보드 UX**: 폼 열림 시 첫 입력 autofocus, Esc 닫기, Cmd/Ctrl+Enter 저장.
- **i18n**: 새 사용자 대면 문자열(툴팁·검증 오류·빈 상태·토스트 등)은 전부 en/ko 카탈로그를 경유한다. 아이콘화로 사라지는 텍스트 버튼의 스모크 셀렉터는 aria-label 기반으로 재작성한다.

## Testing Decisions

- 좋은 테스트는 외부 행동만 본다 — 저장 결과·실요청 헤더·접근성 트리·스토리지 상태를 검증하고, 컴포넌트 내부 구조나 애니메이션 프레임은 검증하지 않는다.
- **vitest (core)**: 필수 필드 검증 순수 함수(종류별 누락 판정, 응답 쿠키 빈 값 허용), ruleView 조건 배지 데이터 확장, 삭제-Undo 원자 복원 명령(Placeholder 규칙의 materialized 값·인덱스가 삭제 전과 동일하게 복원 — plan r1 R-1). 선행 예: rule-summary·materialization 테스트.
- **smoke (Playwright)**: 저장 차단(빈 이름 Save → 오류 표시 + 스토리지 불변), dnd 재정렬 — 마우스 드래그와 **키보드 경로(그립 포커스 → Space 집기 → 화살표 → Space 드롭 → 순서·겹침 승자 반영, plan r1 R-2)** 둘 다(기존 N8 대체), 조건 배지 줄 표시, 아이콘 버튼 aria-label 경로(en/ko), 삭제 Undo(토스트 → Placeholder 값 보존 포함 원위치 복원), 빈 상태 CTA, 환경설정 기본 pill 비제거, 폼 키보드(Esc/Cmd+Enter). 선행 예: 기존 N 시리즈.
- **storybook build**: 교체·신규 프리미티브의 렌더 게이트.
- **ui-diag**: 배지 줄·아이콘화 패널·호버 액션의 라이트/다크 스크린샷과 가로 오버플로 검사.
- **번들 크기 게이트 (plan r1 R-3)**: 기준선 — 현 빌드에서 popup이 로드하는 초기 JS는 popup 엔트리 145B + 공용 청크 386.0KB(min, 비압축), 확장 전체 464KB. 게이트 — motion·dnd-kit 도입 후 popup 초기 JS 청크 합계 증가가 **+60KB(min 기준) 미만**이어야 하며, 초과 시 지연 로드 구조를 재검토하고 재승인 없이는 통과하지 않는다. 측정은 빌드 산출물의 popup 엔트리와 그 import 그래프 청크 크기 합산으로 한다.

## Out of Scope

- 프로필 수준 필터의 부활이나 새 Condition 차원 추가 (ADR 0010 유지)
- Import/Export 공지·오류 문자열의 카탈로그화 (기존 en 하드코딩 패턴 — 별도 과제)
- Firefox 등 다른 브라우저 대응
- 규칙 행의 드래그 재정렬 (이번엔 프로필만)
- 신규 기능 추가 (이 스펙은 상호작용 정제에 한정)

## Further Notes

- Base UI는 1.0.0-rc.0 — API 변동 리스크가 있으나 이미 3개 컴포넌트가 의존 중이며, 일원화로 노출면이 커지는 대신 관리 지점이 하나가 된다 (ADR 0011).
- motion·dnd-kit은 신규 의존성 — 팝업 첫 페인트에 민감하므로 지연 로드(LazyMotion)와 번들 크기 확인을 게이트에 포함한다.
- 문서·코드 전반에서 레퍼런스 제품 브랜드명을 쓰지 않는 기존 원칙을 유지한다.
