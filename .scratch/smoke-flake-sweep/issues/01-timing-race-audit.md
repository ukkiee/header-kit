# 01 — 스모크 타이밍 흔들림 일괄 감사

**Status:** ready-for-agent
**Blocked by:** None

**What to build:** `scripts/smoke.mjs` 전반에서 "반영되기 전에 표본을 한 번만 뜨는"
타이밍 레이스와 "빡빡한 타임아웃"을 훑어 관측 기반(폴링/프레임 수집)으로 바꾼다.
ui-polish 착지 과정에서 스모크가 ~4회에 1번꼴로 간헐 실패함이 드러났고, 세 곳(F1·N23a·I1)은
그 브랜치에서 이미 고쳤다(커밋 `12b7cbe`). 이 이슈는 **나머지**다.

## 배경 — 이미 고친 것 (여기 대상 아님)
- **F1**(탭 도메인): 룰 카운트 대신 헤더 효과를 폴링. 커밋 `12b7cbe`.
- **N23a**(메뉴 순차): rAF 한 번 대신 프레임 수집. 커밋 `12b7cbe`.
- **I1**(자동 백업): 35s → 55s(30s 스로틀 위로 여유 확대). 커밋 `12b7cbe`.

## 남은 대상 (관측된 것 + 감사)
- [ ] **#8 openRuleFormAt / L2g** — `Edit` 버튼이 visible·enabled·stable인데도 클릭이 30s
      타임아웃(드묾, 병렬 실행 중 CPU 경합 냄새). 15회 중 1회. `scripts/smoke.mjs`의
      `openRuleFormAt` 헬퍼. goto 직후 클릭이 액셔너빌리티 재검사 루프에 걸리는 것으로 보임 —
      명시적 대기/재시도 또는 원인 규명 필요.
- [ ] **전 스위트 감사** — 아래 패턴을 훑는다:
  - MutationObserver + rAF **한 번** 표본 (motion 붙기 전에 잡힘) — `firstFrameLabelOpacity`도
    같은 패턴이나 `< 1`만 봐서 현재는 통과. 잠복 위험.
  - `pollUntil`/`waitForTimeout` 상한이 스로틀·전이 길이 대비 빡빡한 곳.
  - `waitFor`가 던지면 스위트가 죽는 경로(값으로 받도록) — 컨덕터 노트에 반복 등장.
- [ ] 수정 후 스모크 **20회 연속 초록**으로 안정 판정.

## 유의
- ui-polish의 게이트·착지와 독립. 이건 하니스 신뢰성 개선이지 제품 변경이 아니다.
- 제품 코드 회귀는 없다 — 전부 관측 시점/타임아웃 문제였다.
