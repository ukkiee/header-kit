# 02 — 스크롤 영역을 앱 스타일로 통일

**What to build:** 사이드바와 본문의 스크롤바가 OS 기본이 아니라 앱의 시각 언어를 따른다(다크 모드 포함). 프로필이 아무리 많아도 팝업이 가로로 넘치지 않고, 규칙이 많은 프로필에서도 스크롤이 부드럽게 동작한다.

## 리뷰 반영 (2축)

- **양 축이 독립으로 짚은 결함:** 한도 재조정 근거로 인용한 시작 지표가 파일마다 달랐다(서로 다른 실행을 인용). 정본을 이 파일 하나로 모으고 나머지는 참조만 하게 했으며, 변동 폭을 명시해 단일 실행을 개선 근거로 쓰지 못하게 했다.
- **Spec: story 9(다크 모드)가 어떤 게이트에도 안 걸렸다** — 토큰에서 `dark:` 변형을 지워도 전부 통과했다. smoke **N22a**(라이트/다크 thumb 색이 다르고 둘 다 불투명)를 추가하고, `dark:` 제거 시 실제로 FAIL하는 것까지 확인했다. **N22b**로 "넘치지 않으면 스크롤바가 렌더되지 않는다"도 못박았다 — 트랙을 기본 노출로 둔 근거가 그것이다.
  - 이 과정에서 선택자 버그를 하나 잡았다. Root에도 `data-has-overflow-y`가 붙어 첫 매치가 **트랙**(투명)이었고, 단언이 조용히 잘못된 요소를 볼 뻔했다. 트랙·thumb을 함께 요구하는 선택자로 좁히고 `count === 1`도 단언에 넣었다.
- **Spec: 스펙 가드라인 문구를 내가 완화했었다** — "한도를 **다시** 올리지 않는다 — **먼저** 지연 로드나…"를 "**먼저** 올리지 않는다"로 바꿔 규칙 자체를 무르게 만들었다. 원문으로 복원하고 재트리아지 사실만 하위 항목으로 덧붙였다.
- **Standards: 두 개의 className prop + Middle Man 래퍼.** `<aside>`/`<main>`이 ScrollArea 하나만 담는 껍데기가 돼 있었다. Base UI `render` 합성(IconButton과 같은 방식)으로 바꿔 랜드마크가 곧 스크롤 컨테이너가 되게 했다 — 래퍼가 사라지고, "호출자가 높이를 확정해야 한다"는 숨은 계약도 그리드 칸이 대신 만족시킨다. 그 계약은 prop 주석에도 명시했다.
- **Standards: 낡은 수치·들여쓰기** — Further Notes의 "여유 3KB"를 현재 값으로, `<main>` 블록 들여쓰기를 정리했다.

## 측정치 (이 파일이 정본 — 다른 문서는 여기를 가리킨다)

- **번들:** ScrollArea **+12.6KB** (global 청크 기준 +116.9 → +129.5KB). 참고로 잰 Autocomplete 비용 **+14.5KB**
  - structure r1 S-1 이후 **계량 기준이 바뀌었다** — 이제 popup이 즉시 로드하는 집합 전체(`global` + `react` + `popup`)를 재고, 현재 **523.8KB = 기준선 386 + 137.8KB (한도 +143KB)**. 위 델타는 여전히 유효하지만(같은 코드가 global에 들어간다), 앞으로의 비교는 즉시 합계 기준이다.
- **시작 지표 (기준선 first paint 64.0ms · dom ready 36.7ms):** 구현 과정에서 5회 측정했고 **중앙값 자체가** first paint **60~68ms**, dom ready **32~49ms** 범위로 움직였다. **회귀 없음** — 기준선과의 차이가 이 변동 폭 안에 있다.

  중요한 관찰: 5회 표본의 중앙값을 써도 실행 간 편차가 first paint ±8ms, dom ready ±17ms다. **단일 실행의 숫자를 개선/악화의 근거로 쓰면 안 된다.** 초기에 이 파일과 다른 문서가 서로 다른 실행을 인용해 4ms 개선처럼 보였고, 리뷰 두 축이 모두 이를 잡아냈다. 이후 수치는 이 파일에만 두고 나머지 문서는 여기를 참조한다.

  이 편차 폭은 티켓 01이 남긴 한계 기록과도 이어진다 — 상한이 `기준선 + 150ms`로 넓게 잡혀 있어 게이트가 성립하는 것이지, 지표 자체가 수십 ms 단위 회귀를 가려낼 만큼 정밀하지는 않다.

## 번들 한도 재트리아지 (+120KB → +135KB)

**어느 하나만 넣어도 기존 한도를 넘었다** — ScrollArea만 +129.5KB, Autocomplete만 +131.4KB. 스펙의 에스컬레이션 사다리를 순서대로 밟은 결과:

1. **지연 로드** — ScrollArea는 사이드바·본문 컨테이너 자체라 첫 페인트에 있어야 한다. 지연시키면 레이아웃 시프트가 생겨 실질적으로 불가.
2. **대안** — 네이티브 스크롤 + CSS 스크롤바는 0KB지만 네이티브 스크롤바는 오버레이가 아니라 스크롤 시 콘텐츠 폭을 6~8px 잠식한다. 스펙의 "폭을 잠식하지 않도록" 결정과 정면 충돌하고 ADR 0011의 Base UI 일원화에서도 벗어난다.
3. **사용자 재트리아지** — 승인받아 한도를 +135KB로 조정.

근거는 추정이 아니라 **실측**이다. 티켓 01이 만든 시작 지표가 ScrollArea 도입 후에도 회귀를 보이지 않았다(위 측정치). 번들 게이트는 체감 속도의 **대리** 지표인데 이제 직접 지표가 있고 둘이 엇갈렸으므로 직접 지표를 채택했다. ui-refine 08이 60→120으로 올릴 때의 논리("로컬 확장이라 네트워크 다운로드 없음")와 같되, 이번엔 숫자가 있다.

**Autocomplete는 지연 로드가 조건으로 티켓 03에 걸렸다** — 규칙 폼 안에서만 쓰이므로 초기 청크에 있을 이유가 없다. 한도를 또 올리는 것은 기본값이 아니다.

## 진단 검사를 조정하지 않은 이유

스펙은 "ScrollArea viewport가 `overflow-x: auto|scroll`에 걸릴 수 있으므로 제외하도록 조정한다"고 했다. 실제로는 **걸리지 않았다** — 이 스캔은 `scrollWidth > clientWidth`를 함께 요구하는데, `ScrollArea.Content`(`min-width: fit-content`)를 쓰지 않기로 해 가로 오버플로 자체가 없다(실측 `inner-scrollers=0`).

예외를 넣었다면 오히려 검사가 약해진다. 세로 스크롤바만 렌더하므로, viewport가 가로 오버플로를 **보이지 않는 스크롤로 조용히 흡수**해도 사용자에겐 아무 단서가 없다 — 지금 검사는 바로 그 상황을 잡는다. 스펙의 전제(조건절 "걸릴 수 있으므로")가 성립하지 않았으므로 예외를 두지 않고, 사유를 스크립트 주석에 남겼다.

## `ScrollArea.Content`를 쓰지 않은 이유

Base UI의 Content 파트는 `min-width: fit-content`를 걸어 내용이 넓으면 컨테이너를 가로로 늘린다. 이 앱의 계약은 정반대다 — 긴 프로필 이름은 잘리고 레이아웃은 유지된다(story 10). children을 Viewport에 직접 둬 그 성질을 들이지 않았다.
