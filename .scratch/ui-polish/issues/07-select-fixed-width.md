## 측정이 설계를 바꿨다 — 고정 폭을 프리미티브 기본값으로 두지 않았다

스펙은 "Select 프리미티브에 폭 제약을 도입하되 최장 라벨이 잘리지 않는 폭을 **기본값**으로"라고 적었다. 그런데 네 개의 Select를 실측해 보니:

| 셀렉트 | 자연 폭 | 내용에 따라 변하나 |
|---|---|---|
| 종류 | 406px | 아니오 — Field가 폭을 정한다 |
| **URL 매치 방식** | **en 132 / ko 94px** | **예 — `shrink-0`으로 같은 행에 있다** |
| 모드 | 199px | 아니오 — 그리드가 정한다 |
| 빈 값 처리 | 199px | 아니오 — 그리드가 정한다 |

**출렁이는 것은 하나뿐이다.** 고정 폭을 기본값으로 두면 나머지 셋이 406·199 → 136px로 좁아진다 — 아무도 요청하지 않은 레이아웃 변경이다. 그래서 `width` 변형(`auto` 기본 / `fixed`)을 두고, 옆을 미는 그 자리에만 `fixed`를 준다. 토큰이 "기본 폭"이라는 스펙의 취지는 그대로다 — 값이 한 곳에 있고 호출자가 고른다.

## 폭 값 8.5rem의 근거

en `Regex (advanced)`가 최장이다: 라벨 102px + 아이콘 12px + 간격 4px + 좌우 패딩 12px ≈ 130px(실측 자연 폭 132px). 여유 4px을 얹어 136px. ko 최장(`정규식 (고급)`)은 64px이라 여유가 넉넉하다.

## 두 방향으로 못박았다

플랜 게이트 R-2가 경고한 것이 "폭 안정성만 보면 폭을 좁게 잡아 라벨이 잘려도 통과한다"였다. 두 퇴화 경로를 각각 되돌려 확인했다.

- `width="fixed"`를 빼면 → 폭 `104.92/72.84/116.34/132.13`, 패턴 좌변 `423.92/391.84/435.34/451.13`으로 **사용자가 보고한 증상 그대로** FAIL
- 토큰을 5rem(80px)으로 좁히면 → 폭은 안정(`80`)이지만 **미절단=false**로 FAIL

## 부수로 고친 것

셀렉트 팝업을 여닫는 동안 Base UI의 inert 백드롭이 다음 클릭을 가로채 스위트가 죽었다. 열림·닫힘을 매번 명시적으로 기다리도록 했다 — `waitForTimeout`으로 눈대중하지 않는다.
