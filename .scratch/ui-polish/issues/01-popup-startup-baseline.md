# 01 — 팝업 시작 성능 계측 + 기준선

**What to build:** 팝업이 이 피처의 변경 **이전에** 얼마나 빨리 뜨는지를 재현 가능하게 재고, 그 값을 기준선으로 못박는다. 이후 티켓이 팝업을 눈에 띄게 느리게 만들면 진단이 실패로 알려준다. 번들 게이트는 바이트만 재므로 한도 안에 있으면서도 첫 페인트가 느려질 수 있다 — 이 티켓이 그 공백을 메운다(스펙 R-4).

이 티켓은 **앱 코드를 바꾸지 않는다.** 기준선의 유효성 전체가 "변경 전 빌드에서 측정했다"에 달려 있다.

**Blocked by:** None — can start immediately.

**Status:** ready-for-agent

- [ ] 진단이 팝업의 첫 렌더와 상호작용 준비를 잰다. 상호작용 준비는 문서 시작부터 규칙 추가 버튼이 존재하고 활성화될 때까지이며, **페이지 내부 시계**로 재 자동화 왕복 지연이 지표에 섞이지 않는다
- [ ] 기존 실데이터를 그대로 쓰고 콜드 로드 5회(매회 새 페이지), 워밍업 1회는 버리고 중앙값을 채택한다 — 같은 커밋에서 두 번 돌려도 판정이 뒤집히지 않는다
- [ ] 기준선을 `docs/reviews/ui-polish/perf-baseline.md`에 기기·중앙값·5회 원값과 함께 기록한다
- [ ] 중앙값이 `기준선 × 1.3`과 `기준선 + 150ms` 중 **큰 값**을 넘으면 진단이 실패(exit 1)한다
- [ ] 기준선 파일이 없으면 측정치만 출력하고 실패시키지 않는다 — 기준선 없는 상태의 거짓 실패를 막는다
- [ ] 기기 의존성과 "같은 기기에서 기준선 대비로만 유효" 한계를 스크립트와 기준선 파일 양쪽에 남긴다
- [ ] `src/` 변경 0 — 기준선이 변경 전 빌드에서 측정됐음이 diff로 확인된다
- [ ] 전 게이트 green
